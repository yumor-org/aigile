# 05. 外部要因および偶発的発見の扱い (δ1, δ2)

**ステータス**: 保留（基本フロー確立後に再開）
**関連**: [escalation.md](../escalation.md)

## 背景

[escalation.md](../escalation.md) の発生タイミング分類で、Phase δ として以下の 2 種が挙げられていますが、現時点では扱いが保留されています:

- **δ1: 外部要因による Document の無効化** — 規制変更、依存パッケージの脆弱性、外部 API の仕様変更などにより、既存 Document が事実上無効になる
- **δ2: 通常作業中の偶発的発見** — レビューや別作業の途中で、既存 Document の矛盾を AI または人間が偶然発見する

これらは「フローの特定タイミング」と紐付かない事象であり、現状のエスカレーション機構（検出元 PR が必須フィールド）にそのまま乗りません。

## 論点

**δ1 と δ2 をエスカレーション機構の中でどう扱うか？それとも別系統として扱うか？**

副次的論点:

- δ1 の検出は誰の責務か（人間？ 定期監査 AI？ 外部システム連携？）
- δ2 のように検出元 PR が存在しない場合、必須フィールド「検出元 PR」をどう代替するか

## 想定シナリオ

### δ1 の例

- **規制変更**: GDPR 改定により、既存 Requirement Document の "ユーザーデータ保持期間" 規定が違法に
- **依存脆弱性**: 使用中の認証ライブラリが廃止され、Architecture Document の認証フローが実装不可能に
- **外部 API 変更**: 連携先 API が breaking change を発表し、Specification Document の振る舞いが実現不可能に

### δ2 の例

- 別 Spec PR をレビュー中の人間が、既存 Requirement Document の別セクションに矛盾を見つける
- AI が定期的な Document 整合性監査を実行し、過去にマージされた Spec と Architecture の不整合を発見
- 新メンバーが Document を読んでいて、過去の議論で見落とされた矛盾に気づく

## 選択肢

### (a) エスカレーション Issue の起票範囲を拡張（人間検出・外部検出も含む）

- 「検出元 PR」フィールドを **任意** に変更
- 代わりに「検出契機」フィールド（PR / 外部要因 / 監査 / 偶発発見）を導入
- すべての Document 欠陥報告を統一フローで扱う
- **メリット**: 単一機構で済む、運用がシンプル
- **デメリット**: エスカレーション Issue の意味がやや拡散する

### (b) "Document 監査 Issue" などの別 Issue 種別を新設

- δ1, δ2 専用の Issue タイプ（例: `audit-issue`）
- エスカレーション Issue と別ライフサイクル、別フィールド
- **メリット**: 各 Issue タイプの意味が明確
- **デメリット**: Issue タイプが増えて全体像が複雑化

### (c) 別フェーズ（定期 Document 監査）として独立した仕組み

- aigile に「Document 監査ワークフロー」を別途用意
- 定期的に AI が全 Document の整合性をチェックし、問題を Issue 化
- δ1（外部要因）は外部システム連携（脆弱性 DB、規制更新通知等）から
- **メリット**: フローの責務が明確に分離
- **デメリット**: 仕組みが増える、aigile のスコープ拡大

### (d) 当面扱わない（手動で Requirement Issue として起票）

- δ1, δ2 を発見した人は、通常の Requirement Issue として「既存 Document の修正要求」を起票
- ただし意味論的には [concepts.md](../concepts.md) の "Requirement Issue = 追加の要求" 定義と微妙にずれる
- **メリット**: 新機構不要
- **デメリット**: 概念モデルの一貫性を犠牲にする

## なぜ現時点で保留か

1. **基本フローの確立が優先**: δ1, δ2 はフロー上の境界事象であり、まず通常フロー（α, β, γ）が確実に回る状態が前提
2. **具体運用が見えない**: 監査の頻度、外部連携の有無、検出の主体などが具体化していない
3. **オーバーエンジニアリングリスク**: 仮想的な需要に対する設計は陳腐化しやすい

## 再開のトリガ

以下のいずれかが具体化したら本論点を再開:

1. 通常フロー（α, β, γ）の運用が安定し、Document 監査の必要性が顕在化
2. プロジェクトが規制対象業界に適用され、外部要因によるエスカレーションが現実的な要件になる
3. Document 規模が大きくなり、偶発的発見の頻度が無視できなくなる

## 暫定方針

保留期間中、δ1, δ2 のような事象が発生した場合は **(d) Requirement Issue として手動起票** で対応し、その実例をアンチパターン/具体ニーズとして蓄積します。十分な事例が集まったら、本論点を再開して (a)〜(c) のどれを採用するか判断します。
