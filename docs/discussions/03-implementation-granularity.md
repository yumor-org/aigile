# 03. 実装 Issue（ステップ 9-10）の粒度ガード

**ステータス**: 未確定
**関連**: [workflow.md](../workflow.md)

## 背景

aigile のワークフローでは、Architecture Document が確定した後、AI が実装計画を立てて **実装 Issue を起票し**（ステップ 9）、その Issue に基づいて **自律的に実装 PR を作成**します（ステップ 10）。

この自律実装フェーズで、AI が暴走しないための **粒度ガード** が必要です。具体的には:

- 1 つの実装 Issue が大きすぎると、AI が広範囲を一度に変更しレビュー困難な PR が生まれる
- 過剰な変更により既存機能が回帰する可能性が高まる
- 親 Requirement との対応関係が見えにくくなる

## 論点

**実装 Issue / 実装 PR の粒度をどう制約するか？**

副次的な論点として、制約をどう強制するか（AI が自律遵守、ツールで自動拒否、レビューゲートで人間判断）も決める必要があります。

## 候補ルール

### 構造ルール

| ルール | 内容 | 目的 |
|---|---|---|
| **R1**: 1 Issue = 1 PR | 1 つの実装 Issue から派生する PR は 1 つに限る | トレーサビリティの単純化 |
| **R2**: 親 Requirement Issue への back-link 必須 | PR 本文に Requirement Issue 番号を必ず記載 | 起点への追跡可能性 |
| **R3**: 既存テスト通過必須 | 既存テストが落ちる PR は自動 Close | 回帰防止 |
| **R4**: 該当 Specification / Architecture セクションへの参照必須 | PR 本文に該当 Document 箇所を記載 | 設計との対応 |

### 数量ルール

| ルール | 内容 | 例 |
|---|---|---|
| **R5**: 最大ファイル変更数 | PR で変更されるファイル数の上限 | 10 ファイル |
| **R6**: 最大変更行数 | PR の +/- 合計行数の上限 | 500 LOC |
| **R7**: 最大ディレクトリスコープ | 単一 PR が触れるトップレベルディレクトリ数 | 2 ディレクトリ |

数量ルールは厳密な閾値設定が難しく、リポジトリ・言語によって妥当な値が異なります。

## 強制方法の選択肢

### (a) AI による自律遵守（ハード閾値なし）

- 粒度ルールはガイドライン文書（例: `.aigile/implementation-policy.md`）として記述
- AI はこのガイドラインを読んで実装計画を立てる
- **メリット**: 柔軟、状況対応可能
- **デメリット**: AI の判断ミスを直接ブロックできない

### (b) ツール側でハード閾値による自動拒否

- aigile ツールが PR の数量を測定し、閾値超過なら自動的に PR を Close（または Block ラベル）
- **メリット**: 確実な強制
- **デメリット**: 妥当な大きな変更も拒否される、閾値調整が難しい

### (c) ソフト閾値 + 人間レビュー強制

- 閾値超過時は人間承認を必須化（`stakeholders.yml` の AI 承認をバイパス）
- **メリット**: バランス型、AI の判断を尊重しつつ大きな変更には人を介在
- **デメリット**: ツール側のロジックが複雑化

### (d) 何も設けない（リポジトリの CODEOWNERS / ブランチ保護に委ねる）

- aigile としては粒度ガードを定義せず、GitHub ネイティブの仕組みに任せる
- **メリット**: aigile の責務を狭く保てる
- **デメリット**: aigile が「AI 自律実装」を売りにしている以上、無防備すぎる

## トレードオフ比較

| 強制方法 | 確実性 | 柔軟性 | 実装複雑度 | aigile らしさ |
|---|---|---|---|---|
| (a) AI 自律遵守 | △ | ◎ | 低 | ○ |
| (b) ハード閾値拒否 | ◎ | × | 中 | △ |
| (c) ソフト閾値で人間レビュー強制 | ○ | ○ | 高 | ◎ |
| (d) 何も設けない | × | ◎ | 低 | × |

## 暫定推奨

**構造ルール (R1-R4) は厳格に、数量ルール (R5-R7) はソフト閾値で人間レビュー強制 (c)** を推奨します。

理由:

- 構造ルール（1 Issue 1 PR、back-link、テスト通過、Document 参照）は **トレーサビリティの根幹** であり、aigile の主目的そのもの。これは違反した PR を自動拒否しても運用上の問題はない
- 数量ルールは絶対的な正解がないため、超過したら **人間判断を強制** することで安全装置にする

設定ファイル案:

```yaml
# .aigile/implementation-policy.yml
structural:
  one_issue_per_pr: true              # R1
  require_requirement_backlink: true   # R2
  require_existing_tests_pass: true    # R3
  require_doc_reference: true          # R4

soft_thresholds:
  max_files_changed: 10                # R5: 超過時は人間レビュー強制
  max_lines_changed: 500               # R6
  max_directory_scope: 2               # R7
```

- 実装 Issue テンプレートの策定時
